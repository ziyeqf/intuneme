# MkDocs Documentation Site for intuneme

**Date:** 2026-03-23
**Status:** Draft

## Overview

Add a MkDocs documentation site using Material for MkDocs (MaterialX) to intuneme, replacing the monolithic README with a structured, multi-page docs site deployed to GitHub Pages. CLI reference pages are auto-generated by a new `intuneme gendocs` command and built at deploy time (not committed).

## Goals

- Professional, navigable documentation site consistent with the frostyard brand (matching yeti's frost theme)
- Content organized by user journey: Getting Started → User Guide → Troubleshooting → Reference
- CLI reference always in sync with the code (generated at build time, never stale)
- Minimal README that points to the docs site

## Non-Goals

- Versioned documentation (single latest version is sufficient)
- Custom MkDocs plugins
- Migrating internal planning docs (`docs/plans/`, `docs/superpowers/`) into the site

## 1. `gendocs` Command

### Implementation

New file: `cmd/gendocs.go`

- Registers `gendocs` command via `init()` with `rootCmd.AddCommand()`
- Takes one required positional argument: output directory path
- Uses `github.com/spf13/cobra/doc.GenMarkdownTree(rootCmd, outputDir)` to generate one `.md` file per command
- Creates the output directory if it doesn't exist
- The `gendocs` command itself should be hidden (`cmd.Hidden = true`) so it doesn't appear in user-facing help or generate a reference page for itself
- Set `rootCmd.DisableAutoGenTag = true` before generating — prevents a date-stamped footer (`Auto generated by spf13/cobra on ...`) that would differ on every CI run

### Output

One markdown file per non-hidden command. Expected output includes:
- `intuneme.md` (root command overview)
- `intuneme_init.md`
- `intuneme_start.md`
- `intuneme_stop.md`
- `intuneme_destroy.md`
- `intuneme_shell.md`
- `intuneme_status.md`
- `intuneme_config.md`
- `intuneme_config_broker-proxy.md`, `intuneme_config_broker-proxy_enable.md`, `intuneme_config_broker-proxy_disable.md`
- `intuneme_open.md`, `intuneme_open_edge.md`, `intuneme_open_portal.md`
- `intuneme_recreate.md`
- `intuneme_udev.md`, `intuneme_udev_install.md`, `intuneme_udev_remove.md`
- `intuneme_broker-proxy.md` (top-level proxy runner, distinct from `config broker-proxy`)
- `intuneme_extension.md`, `intuneme_extension_install.md`
- `intuneme_completion.md` and per-shell variants (`_bash`, `_zsh`, `_fish`, `_powershell`) — Cobra auto-registers this command; the generated pages are useful reference
- `intuneme_help.md` — Cobra's default help command; generated automatically

Note: `broker-proxy` contains a hyphen, which cobra/doc warns about for man page section numbering. This does not affect markdown generation — filenames and cross-links work correctly.

### Makefile Target

```makefile
gendocs: build
	@echo "Generating CLI reference..."
	@mkdir -p site/reference/cli
	@./$(BINARY_NAME) gendocs site/reference/cli/
```

Update the existing `docs` target:
```makefile
docs: completions manpages gendocs
```

### Dependency

`cobra/doc` is a subpackage of `github.com/spf13/cobra` which is already a dependency. Only the import is needed — no new module dependency.

### Nav Note on CLI Commands

The nav lists only the root CLI command page (`reference/cli/intuneme.md`). Cobra's generated markdown includes "SEE ALSO" links to subcommand pages, so users navigate from the root page into individual commands naturally. This avoids hardcoding every command name in `mkdocs.yml` (which would require updating the nav every time a command is added).

MkDocs will log warnings for generated pages not listed in `nav` — this is expected and harmless. The pages are still built and reachable via links.

## 5. GitHub Actions Workflow

File: `.github/workflows/docs.yml`

```yaml
name: Deploy docs

on:
  push:
    branches: [main]
    paths:
      - "site/**"
      - "mkdocs.yml"
      - "cmd/**"
      - "internal/**"
      - ".github/workflows/docs.yml"
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: true

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0   # needed for git describe (version ldflags)
      - uses: actions/setup-go@v5
        with:
          go-version-file: go.mod
      - run: make gendocs
      - uses: actions/setup-python@v5
        with:
          python-version: "3.x"
      - run: pip install mkdocs-materialx
      - run: mkdocs build
      - uses: actions/upload-pages-artifact@v3
        with:
          path: _site
      - uses: actions/deploy-pages@v4
        id: deployment
```

### Trigger Paths

The workflow triggers on `cmd/**` and `internal/**` changes in addition to `site/**` and `mkdocs.yml`. This ensures CLI reference pages are regenerated when command help text changes, even if no docs source files changed.
